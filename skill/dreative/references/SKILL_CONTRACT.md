# Specialist contract

UX and mobile apply to every page. Other specialties are selected internally
because they serve the chosen concept; they are not user-facing approval
controls.

A selected specialty must create a perceptible improvement in a named part of
the real product. It must survive desktop and mobile translation and may not
break content, navigation, accessibility, or core behavior.

Treat user-selected specialties and tier promises as binding delivery scope.
Do not silently omit, weaken, or substitute them. If product evidence creates
material uncertainty about where or how an intense specialty belongs, ask the
user a focused question before changing the agreed allocation.

Use the fewest specialties needed. One strong mechanism may serve several
specialties. Do not add a second runtime or visual system merely to increase
coverage.

Fallbacks are engineering decisions. Use them when capability or measured
performance requires it, preserve the concept, and disclose material changes
before implementation when possible and in the handoff. A fallback that loses
the selected specialty's perceptible contribution is incomplete, not complete.
